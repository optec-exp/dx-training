01
流程设计能力 — 把"麻烦的工作"翻译成系统语言
+

这是本课程最核心的能力。不需要懂代码，但需要能把业务流程描述成：

触发条件 → 处理步骤 → 结果输出

实际例子（作品37）：

客户发邮件
→ 接单员填写网页表单
→ Kintone 自动登录案件记录
→ Google Drive 自动创建案件文件夹
→ Slack @通知负责人

以前这个流程需要手动操作3个系统，现在填一次表单全部自动完成。

练习方法：遇到重复性手工操作，先画出流程图，再用这个三段式描述，然后告诉 Claude Code 去实现。

02
环境变量 — 系统的"钥匙箱"，永远不能写在代码里
+

什么是环境变量？

就像公司的门禁卡不能贴在门上一样，API密钥、Webhook URL 等"钥匙"不能写在代码里，要单独存放在 .env.local 文件中。

.env.local 文件长这样：

SLACK_WEBHOOK_URL=https://hooks.slack.com/services/...
KINTONE_API_TOKEN=xxxxxxxxxxxxxxxxxx
GOOGLE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----..."

为什么重要？

✅ 代码上传到 GitHub 时，.env.local 不会被上传（安全）
✅ 换系统/换频道时，只需修改这个文件，不用动代码
✅ 每个作品各自独立管理，互不干扰

今天的实际操作：Slack Webhook URL 更换 → 只需修改4个作品各自的 .env.local，代码完全不用动。

03
数据存放位置 — 什么数据应该放在哪里
+

不同的系统适合存不同类型的数据，混用会造成管理混乱：

系统适合存什么特点
Kintone案件、客户、AWB 等业务核心数据全公司共享，有权限管理
Supabase需要实时同步、AI分析的数据云端数据库，实时监听变化
Google Drive文档、PDF、报告书、附件文件存储，支持共享
Google Sheets需要手动编辑的表格数据表格式，可直接查看编辑
.env.localAPI密钥、Webhook URL 等敏感信息只在本地，不上传
04
API 的概念 — 系统之间传数据的"管道"
+

什么是 API？

API（Application Programming Interface）= 两个系统之间约定好的"传话方式"。

类比：餐厅里，你（应用）告诉服务员（API）要什么菜，服务员去厨房（另一个系统）取餐，再端给你。你不需要知道厨房怎么做菜。

本课程用到的 API：

API做什么
Kintone API读取/写入/更新 Kintone 里的记录
Google Drive API自动创建文件夹、上传文件
Google Docs API填充模板、生成 PDF
Slack Webhook向指定频道发送消息
汇率/天气 API获取实时汇率、天气数据
Claude API发送文本给 AI，获取分析结果
05
本地 vs 公网 — localhost 和 Vercel 的区别
+

这是很多人容易忽略的概念，但在配置 Kintone Webhook 时非常关键。

状态地址谁能访问用途
本地开发localhost:3000只有自己电脑开发测试
ngrokhttps://abc.ngrok.io全网可访问临时测试 Webhook
Vercel 部署https://xxx.vercel.app全网可访问正式上线使用
作品48 的情况：
Kintone 知识库 App 765 新增记录时，需要主动"推送"通知到我们的应用。
但如果应用只在 localhost:3000，Kintone 根本访问不到这个地址。
→ 所以需要 ngrok（测试用）或 Vercel（正式用）提供公网 URL。
06
AI 提示词设计 — 业务知识是最大的武器
+

Claude API 的效果好不好，70% 取决于提示词写得好不好。而写好提示词最需要的，不是编程知识，而是业务知识。

好的提示词 vs 坏的提示词：

❌ 差：「请分析这个案件的风险」

✅ 好：「你是物流案件风险分析师。请根据以下信息判断延迟风险等级（高/中/低），并给出具体原因：案件番号、AWB番号、当前状态、ETA、航班信息。如果是高风险，请同时给出应对建议。」

关键原则：

1. 给 AI 定角色（你是XXX专家）
2. 给业务背景（越具体越好）
3. 指定输出格式（列表/JSON/高中低三档）
4. 给例子（Few-shot，让 AI 照样做）

你的业务知识（航空货运的流程、风险点、术语）是 AI 输出高质量结果的关键，这是编程人员无法替代的。

07
验证测试思维 — "做完"不等于"做好"
+

每次完成一个功能，必须按以下步骤验证：

验证清单：

✅ 正常情况：提交表单 → 数据在 Kintone 里出现了吗？
✅ 通知验证：Slack 频道收到了正确的消息吗？
✅ 数据验证：字段内容是否正确，没有乱码？
✅ 错误处理：填错数据时，页面有没有提示？
✅ 边界情况：空白提交、重复提交会怎样？

不会写代码没关系，但必须能判断结果是否正确。这就是业务人员做开发最重要的价值——你知道"正确的结果应该是什么样的"。
