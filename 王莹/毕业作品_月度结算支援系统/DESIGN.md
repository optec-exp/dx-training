# 毕业作品 · 月度结算支援系统 — 设计定稿（DESIGN.md）

> 货代（空运为主、海运为辅 + EC）月度结算 / 对账 / 决算 的支援系统。
> 毕业作品先行，但**架构按生产级设计、不返工**。

---

## 1. 项目定位

| 项 | 说明 |
|---|---|
| 性质 | 王莹毕业作品；目标可平滑升级为生产级系统 |
| 核心价值 | 把"人肉比对 Kintone 数据 vs PDF/Excel 账单"的对账，变成"AI 解析账单→结构化→自动比对" |
| AI | 毕业阶段用**免费 Gemini**（多模态读 PDF/Excel）；将来换 **Claude API** |
| 铁律 | **Kintone 全程只读，绝不回写任何数据**；差异**只标记**，由业务财务复核 |

```

        │ REST API（手动触发同步为快照）
        ▼

        │
```

## 3. 核心业务规则（设计的地基）

### 3.1 双口径（必须分开，绝不混用）
| 用途 | 口径 | 时间基准 |
|---|---|---|
| 利润汇总 | 权责发生制 | **纳品完了日** 所在月 |
| 银行决算勾稽 | 收付实现制（现金） | **实际收付款日** |

### 3.2 双法人 + 多 App 架构
**法人一 OPTEC EXPRESS（空运+海运）**：AIR 案件管理 / SEA 案件管理 / EXP 请求入金（AIR·SEA 共用）/ EXP 支付（AIR·SEA 共用）/ 贩管费（日本）/ 中国贩管费
**法人二 OPTEC TRADING（EC）**：EC 案件管理 / EC 请求入金 / EC 支付 / EC 贩管费
**通用**：银行残高管理（两公司共用）

- 同类 App **合并**为统一镜像表，加 `company`(EXPRESS/TRADING) + `business_line`(AIR/SEA/EC) + `source_app` 标识列。
- 对账/同步/决算**按公司边界路由**：AIR·SEA 案件 ↔ EXP 请求/支付；EC 案件 ↔ EC 请求/支付。

### 3.3 OPT 编号 = 唯一关联键，但存在两层 1:N
```
案件(OPT编号)
  └─1:N→ 账单(账单编号 OPTxxx-1 / OPTxxx-2 …)   多收款人/分单；支付侧=多供应商
            └─1:N→ 入金 / 出金记录              同一账单分批收付款
```
→ 所有金额比对统一"**先按 OPT(+供应商) 聚合，再比对**"。
> OPT 编号**跨公司全局唯一**；但两公司数据需**各自独立**呈现/关账/汇总，故全程保留 company 维度做分区与过滤。

### 3.4 成本与账单
- Kintone 成本按 **供应商 / 费用科目分行**；基本 1 账单=1 行，偶有多行 → 按 `OPT+供应商` 汇总比对。
- **对账用原币种**（Kintone 已换算的日元/人民币仅供毛利核算，对账不用）。
- 成本账单为 **PDF/Excel，未进 Kintone** → 对账靠人肉的根因，也是 AI 发力点。
- 匹配键：90%+ 账单带 `OPT编号 + 提单号` → 精准匹配；SOA 部分明细无 OPT → 用**提单号(MAWB/HAWB)** 兜底匹配；仍无法匹配 → 人工兜底。

### 3.5 关账规则
- **硬性规则：拿全所有成本才能关账** → 用"缺账单清单 + 关账门禁（齐全率 100%）"，**不做预提/冲销**。
- 差异**只标记不自动改**：差异 → 业务财务复核 → 账单错则要求代理改单。

### 3.6 出金来源（已封闭）
业务相关入金/出金 = 请求入金 App（EXP/EC）/ 支付管理 App（EXP/EC）；非业务相关 = 贩管费 App（日本/中国/EC）。

## 5. 六大模块（P0 核心）

### 模块① 成本账单 AI 解析入库
- 网页上传 PDF/Excel → Storage → Gemini 多模态 + responseSchema → `bill_lines`。AI 只解析、不判断匹配。
- 单票账单量大 → 演示用样本；生产换 Claude API。
> 贩管费维持人工录入 Kintone（流水自动化已取消：拿不到费用归属月、且部署为多部门联动录入，强行自动化易错）。

### 模块② 对账（单票 + SOA）
```
1. 两侧按 opt_no + 供应商 + 原币种 聚合（按 company 路由）：
   账单侧=SUM(bill_lines)；Kintone侧=SUM(kc_payments 该OPT该供应商)
2. 同币种 + 容差比对 → reconciliations：
   ✅匹配→已解决  ⚠️金额不一致→待复核  🟡Kintone有/账单无→缺账单(计入门禁)
   🔴账单有/Kintone无→漏录/同步异常  ❓账单无opt_no→提单号兜底→否则供应商总额核对+人工
3. SOA(1:多票) 拆行后逐行走同一逻辑
```
**重复检测**（货代常见钱损点）：同一票被代理重复开票、内部重复付款 → 按 `供应商+票号+金额+币种` 近似检测重复账单/重复出金，标记 `duplicate_flags` 待复核。

### 模块③ 缺账单门禁 + 关账/锁账流程
- **缺账单门禁**：该利润月每条成本行须"有匹配账单且无未决差异"；齐全率=100% 才允许月结（按 company 分别关账）。
- **关账状态机**：`进行中 → 月结(软关账, 齐全率100%, 可出月报) → 正式锁账(硬锁)`。
  - **宽限期**：M 月案件在 **M+2 月 1 日**正式锁账（例：5 月→7 月 1 日，宽限 1 个月）。锁账日做成可配置规则。
  - **正式锁账后**：该期间**快照冻结**（period_snapshots），报表可复现、数据不可改。
  - **解锁**：锁后修改需走解锁流程，记录申请人/理由/时间（unlock_logs）；解锁→改→再锁，全程审计。
- **记录级锁定**：日常任一条"已确认"的收入/成本要改也需解锁（源数据在 Kintone，本系统通过再同步**比对快照差异**侦测锁后变更并告警）。

### 模块④ 三 App 同步排查（逐行 by OPT，可定位差异）
```
按 company + opt_no 逐票：
  收入：kc_cases.收入 ⟷ SUM(kc_receipts 该OPT)
  成本：kc_cases.成本 ⟷ SUM(kc_payments 该OPT)
不一致 → sync_checks（两边金额+差额）；另给当月总额概览
```

### 模块⑤ 多维利润汇总 + 毛利率审查
- **9 维度**：对应小组 / 服务类型 / 国别 / 顾客 / Business Scope / 业务范围 / Mode / 出发 / 到达。
- **按分利润**：仅"对应小组"维度用按分（复用马拉松「月度利润分配」逻辑）；其余 8 维直接按案件毛利汇总。
- **审查指标 = 加成率（Markup Rate）= 利润 / 成本 = (收入−成本)/成本**（非毛利率）。
- **加成率计算**：单票加成率=(收入−成本)/成本；月度平均加成率按大类/(大类×服务类型) 汇总。**全部案件都算加成率与平均**。
- **加成率审查标准表（Business Scope × 服务类型，业务规定值，直接采用不自行换算）**：
  | 大类 | ECO | NFO | OBC（按提货时间） | 其他 |
  |---|---|---|---|---|
  | Aerospace | 40% | AOG·NFO 55% | AOG·OBC：6h内提货 150% / 6h后 100% | Parts Procurement 35% |
  | Ship Spares | 40% | 45% | 6h内提货 150% / 6h后 100% | — |
  | Other | 45% | 55% | 6h内提货 150% / 6h后 100% | — |
  > 全部为**单值目标**。OBC 目标由"**是否 6 小时内提货**"决定（150% / 100%）。
- **审查规则**：**仅对落在标准表范围内的案件比对**（范围外仍算加成率，但不审查）。**全部为单值目标**：
  - 实际加成率 **相对目标偏离超过 ±X%**（相对比例；X 可配置，默认 10%）（高/低双向）→ 标记审查。
    例：目标 40%、X=10% → 正常 36%~44%。
  - **OBC 取目标前**先判定案件**是否 6 小时内提货** → 6h 内取 150%、6h 后取 100%，再按上式比对。
- 标准表做成**可配置**；聚合落 `profit_snapshots`，审查结果落 `markup_reviews`。
- **预实差异分析**（本次纳入 P0）：实绩 vs 预算（毛利/贩管费/净利）逐项算差异额、达成率，超预算/未达标自动标红，按 报表对象×周期 展示。

### 模块⑥ 月度决算 · 银行残高勾稽
> 数据限制：入出金记录**不带"银行"字段** → 无法做"银行×币种"，只能做**按币种**勾稽。
```
按 币种（跨所有银行、跨两公司汇总）：
  残高差额 = SUM(kc_bank_balance.残高差额 该币种, 跨所有银行)
  现金净额 = SUM(kc_receipts 入金 该币种) − [ SUM(kc_payments 出金) + SUM(kc_pankanfei 出金) 该币种 ]
  差异 = 残高差额 − 现金净额  → 仅提示（不阻断关账）
口径 = 实际收付款日（现金口径）
```

### 模块⑦ 资金管理（本次纳入 P0）
- **应收/应付账龄**：按客户/供应商列未收/未付，按账龄分桶（0–30/31–60/61–90/90+），超期预警。
- **资金/现金流预测**：基于应收应付预计收付日，滚动预测现金净流入/流出。
- **客户信用额度**：客户授信额度 vs 已用（未收应收）占用率，超限预警。
- **闲置资金投资台账**：投资品种/金额/收益率/到期日/流动性可视化；**与现金流预测联动**（可投额度·留存流动性·回笼时点）。数据源待定（手工/Kintone App）。

### 模块⑧ 风控异常检测面板
统一看板汇集：负毛利票 / 加成率超标 / 异常大额 / 重复账单·重复付款 / 长期挂账（呆账） / 海外代理差异趋势。每项可下钻到单票。

### 模块⑨ AI 洞察 + 经营汇报
- **月报经营点评**：复用"代码算统计、AI 解读"模式，自动生成中日双语经营点评（毛利/净利/预实差异/异常摘要）。
- **各部门月度经营汇报自动生成**：**月度 + 累计(YTD)** 双口径，按部门/全社生成中日双语汇报；输出网页 + **导出 Doc / Slack 推送**（复用作品 41 Google Docs、作品 30 Slack）。
- **对账差异疑因建议**：AI 对每条差异给出可能原因（汇率/科目错配/重复/漏录）辅助复核。
- 免费 Gemini → 未来 Claude API。

### 跨模块 · 内控基座
- **审计日志（audit_logs）**：本系统内所有差异裁决、复核、关账、解锁、预算/人数录入均留痕（谁/何时/改了什么）。
- **快照冻结（period_snapshots）**：正式锁账时冻结该期聚合，保证历史报表可复现。

## 5.5 利润报表与可视化设计

### 报表矩阵（周期 = 年度 / 月度）
| 报表对象 | 指标 | 维度 | 预算 |
|---|---|---|---|
| 全社 | 毛利 + 贩管费 + 净利 | 9 维度 | ✅ |
| 中国 | 毛利 + 贩管费 + 净利 | — | ✅ |
| 日本 | 毛利 + 贩管费 + 净利 | — | ✅ |
| 小组 | 毛利 + 贩管费 + 净利 | 4 维度（国别 / 输入 / 输出 / 見積） | ✅（无役員関連費用）|

- **净利**：业务小组净利 = 该小组毛利 − 该小组自身贩管费；聚合层（全社/中/日）净利 = 毛利合计 − **全部**贩管费（含管理部门）。**税金、对象外除外**，不进净利。
- 可视化：柱状/饼图（毛利·贩管费·净利 按维度）、加成率红绿灯审查、预算 vs 实绩对比。
- 两法人/地域各自独立视图。
